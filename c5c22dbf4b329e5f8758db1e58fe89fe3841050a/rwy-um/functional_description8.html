Call forwarding check at registration

If this optional feature is implemented and activated (IsCFRegistrationForbidden is set to true), at stage 2 of the registration (see above), checks whether the MSISDN has activated call forwarding. If yes, the registration is rejected. If no, the registration procedure is continued as described above.

Note: Only unconditional call forwarding is checked.

In order to check whether the MSISDN has activated call forwarding, the system sends an AnyTimeSubscriptionInterrogation (ATSI) message which contains a requestedSubscriptionInfo.ssInfo.ssCode.forwarding parameter to the HLR. The HLR returns an ATSI Response (ATSIR) with the callForwardingData.forwardingFeatureList[] parameter which is evaluated by the system. In the case of an error during the communication with the HLR, a confirmation ticket is written and the registration is continued as described above.

Bulk registration

When the system detects an incoming registration request, it checks whether the request contains a bulk tag (BULK_TAG). If yes, it checks whether bulk operations are allowed (IsBRAllowed), whether the request is correctly formatted, whether it was sent by a lead driver (FC=01) and whether the Cabradio Flag is set to true for the registering MSISDN. Only if all checks are successful, the registration is performed (separately for each FC as described for normal registration).

If other features interworking with an external system are implemented as well (e.g., external databases connected over an online interface, or another system in the case of geographical redundancy), one bulk operation will cause up to 10 single messages to each of the external systems.

For bulk registration, the following handlings are supported:

  • The cab radio or the lead driver sends a registration request message containing the list of FCs which they want to register for.

    The system processes the request and returns a response message, that is, a list containing the successfully registered FCs and the not successfully registered FCs.

  • The system can override the list included in the message with an own list of FCs.

    The FC lists are administrated in the BulkRegistration data set. For each User Identifier Number (UserIdentifierNumber), separate lists for the call types CT2 (FC_LIST_CT2), CT3 (FC_LIST_CT3) and CT4 (FC_LIST_CT4) can be administrated.

Note: If bulk operations are to be used by the cab radio, it must be configured to allow network override.

The response message sent by the system has the following format:

<OC> <NUM_SFC> <FC_SLIST> <NUM_UFC> <FC_ULIST>

Table 2. Bulk registration
Placeholder Description
OC Operation code (outcome) ** for bulk registration
NUM_SFC Number of successfully registered FCs
FC_SLIST List of successfully registered FCs
NUM_UFC Number of unsuccessfully registered FCs
FC_ULIST List of unsuccessfully registered FCs
Note: The list of registered FCs in the response message differs from the originally sent list of FCs if a list of FCs was administrated which overrides the sent FCs.
