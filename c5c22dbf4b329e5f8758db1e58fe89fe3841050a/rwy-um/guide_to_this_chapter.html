Guide to this Chapter

Guide to this chapter

This chapter describes the features of the service. These features comprise the following subjects:

  • Access to the service
  • Functional addressing
  • Functional address update
  • Location-Dependent Addressing (LDA)
  • User self-administration via USSD (i.e. registration, deregistration and interrogation)
  • Dispatcher terminal functions
  • Dynamic creation of data sets
  • Authorization checks using access matrix
  • USSD notification of users via the Flexible User Messaging (FUM) feature
  • Service Management tasks
  • Dynamic voice mailboxes
  • Acknowledgement center
  • N-based voice group call
  • On-board functionality
  • Geographical redundancy
  • enhanced Railway Emergency Call (eREC)

Access to the service

Four access variants, described in section Access to the Service, can be distinguished for the service:

  • MOC access
  • USSD access
  • SMS access
  • DATA access
  • ESX access

Functional addressing

Functional addressing, described in the section Functional Addressing, means that the railway staff can be reached via their current function rather than via their MSISDN.

Functional address update

If a mobile user registers his MSISDN to a Functional Number (FN)/deregisters his MSISDN from an FN, the dispatcher system/Controller Server is updated with the corresponding registration/deregistration information, covered in Update of Functional Address.

Location-Dependent Addressing

Location-Dependent Addressing (LDA), covered in Location-Dependent Addressing and Update of Train Location Information, means that calls are routed depending on the current position of the train or mobile.

USSD administration

Users can perform certain self-administration tasks via USSD (see Registration, Deregistration and Interrogation.

Dispatcher terminal functions

A dispatcher terminal (see sections Registration and Deregistration of Dispatcher Terminals, Change of Dispatcher Terminal Registration, and Forced Deregistration Initiated by a Dispatcher Terminal:

  • Functional Number (FN) registration
  • FN deregistration
  • Change registration information in the IN database
  • Forced deregistration

Dynamic creation of data sets

Necessary data which cannot be administrated in advance can be created at runtime, described in Dynamic Creation of Data Sets.

Enhanced access matrix

Whether a call is allowed is determined via an access matrix, described in Authorization Checks via Access Matrix.

FUM feature

The users are notified about the outcome of their registration attempts via the FUM feature described in Flexible User Messaging.

Service Management task

A DataBase Administration, covered in DataBase Administration (DBA), is a feature which performs garbage collection operations on a large number of data sets.

Dynamic voice mailboxes

If a user registers his MSISDN to an FN, a dynamic voice mailbox is assigned to the FN (see Dynamic Voice Mailboxes for Functional Numbers).

Acknowledgement center

The acknowledgement center, described in Acknowledgement Center, writes special tickets after an emergency call has occurred.

IN-based voice group call

A mobile dispatcher having registered an FN with Call Type 7 can only perform a group call, if the IN-based Voice Group Call feature, covered in IN-based Voice Group Call, is activated.

On-Board Functionality

The On-Board Functionality (OBF) enables users registered on the same train to communicate with each other using defined short codes (see On-Board Functionality).

Geographical redundancy

Geographical redundancy, described in Geographical Redundancy (GeoRed), means that the GSM-R system encompasses two identical IN platforms each containing a GSM-R service to ensure availability in catastrophic scenarios.
