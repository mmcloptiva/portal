Structure of the Service

Introduction

This chapter describes the way service data of the service are organized using a relational database model.

In comparison to older service versions, currently data are stored in a flat and relational database. This solution makes administration easier and boosts all data-related operations. Relations between specific data sets eliminate the need for data redundancy, and allow for a more user friendly GUI administration.

Relations

Relational database model creates a virtual ‘overlap’ of data. This creates certain constrains while administering overlapped fields - if a field is in a relation with field from other data set, user will be allowed to create or update entries using only values from the related field (this will be executed via drop-down lists). This solution prevents from improper administration of data, ensuring that all relations are preserved.

Views

Views are a new way of data overview. A View presents a combination of columns from different data sets, filtered entries of a single data set etc. This allows for an easy way to gather basic information about data stored in service (e.g. Subscriber registrations view displays all current registrations of all subscribers).

Table 1. Service views and description
View Description
Subscriber registrations View Contains information from Function Register data set about subscribers that are currently registered to Functional Numbers.
Static VoiceMailbox View Contains information about subscribers and static voice mailboxes that are attributed to them.
Dynamic VoiceMailbox View Contains information about subscribers and dynamic voice mailboxes that are attributed to them.
