Functional Description

Introduction

This section describes the basic mechanism to change an existing registration of an MSISDN and an FN of CT 7 via dispatcher terminal.

The feature can also be used to register the MSISDN of a dispatcher terminal to an FN of CT 7 which has not been registered yet.

Messages of the iMGW

The following messages are sent via iMGW:

DTFNStatus.Req (request message):

This message is sent from the CS to the SEP to change the existing registration of the dispatcher terminal. The message includes

  • the FN of the existing registration
  • the ISDNNo for changing the existing registration

DTFNStatus.Resp (response message):

If the change of the registration was successful, this message is sent from the SEP to the CS to answer the DTFNStatus.Req message.

DTFNStatus.Rej (reject message with reject reason):

If the change of the registration was not successful, this message is sent from the SEP to the CS to answer the DTFNStatus.Req message. It includes the reason for reject.

DTS handling

The following list provides an overview of how the DTS feature is handled:

  1. Feature activation
  2. MSISDN check
  3. Authorization check
  4. Assignment limit check
  5. FN check
  6. Mailbox check
  7. Confirmation of registration change

Feature activation

An existing registration can only be changed, if the DTS feature is activated (Is Dispatcher Terminal Status Allowed field set to true). If this is the case, the CS sends the DTFNStatus.Req message to the SEP. After the SEP has received the message, the registration process is started.

MSISDN check

checks whether there is a record in the Subscriber data set for the MSISDN of the user:

  • If yes, the registration process is continued.
  • If not, a confirmation ticket with appropriate event ID is written and the DTFNStatus.Rej message is sent to the CS. The message includes the reason for reject.

Authorization check

If the Is Authorization Checked flag is set to true for the subscriber, the following authorization check is performed: It is checked whether the user is allowed to register for the respective CT (Function Permission.Call Type):

  • If a match for the user’s CT is found in the Function Permission.Call Type field, the registration procedure is continued.
  • If no match is found, a confirmation ticket with appropriate event ID is written and the DTFN-Status.Rej message is sent to the CS.

If the Is Authorization Checked flag is set to false, the authorization check is not performed.

Assignment limit check

An MSISDN may have a maximum number of FN assignments. This registration limit is defined in the Max Function Registered By User field. An MSISDN may only be assigned to an FN, if the maximum number of registrations has not been reached. Otherwise, a confirmation ticket with appropriate event ID is written and the DTFNStatus.Rej message is sent to the CS.

FN check

The service checks whether there is an MSISDN assigned to the FN, and if so, which state the FN is in. The following scenarios are possible:

Table 1. FN check scenarios
If Then
the FN specified in the DTFNStatus.Req message does not exist in the database, a confirmation ticket with appropriate event ID is written and the DTFNStatus.Rej message is sent to the CS.
there already is a valid assignment of another MSISDN to the FN specified in the DTFNStatus.Req message, the service deletes the assignment and stores the new MSISDN in Msisdn in Function Register and sets a new expiration date (if Validity Period is >0).
there is an expired assignment of another MSISDN to the FN specified in the DTFNStatus.Req message, the service deletes the assignment and stores the new MSISDN in Msisdn in Function Register and sets a new expiration date (if Validity Period is >0).
the MSISDN specified in the DTFNStatus.Req message is already registered to the FN specified in the DTFNStatus.Req message (assignment expired or still valid), the service renews the registration and postpones the expiration date by Validity Period (if Validity Period is >0).
no MSISDN is assigned to the FN specified in the DTFNStatus.Req message, the service stores the new MSISDN in Msisdn in Function Register and sets an appropriate expiration date (if Validity Period is >0).

Mailbox check

If a call to the FN was routed to the corresponding mailbox, while the FN was not assigned to an MSISDN, the user is informed about this via a USSD message.

Confirmation of registration change If the registration change was successful, i.e., if the MSISDN of the registration has been replaced by the MSISDN specified in the request message, the DTFNStatus.Resp message with reason 0 (registration change successful) is sent to the CS.
