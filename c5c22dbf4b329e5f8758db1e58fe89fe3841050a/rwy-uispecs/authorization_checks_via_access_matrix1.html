Authorization Checks via Access Matrix

Introduction

This section provides a detailed functional description of how the service handles call authorization checks via an access matrix.

For a description of the feature, see Authorization Checks via Access Matrix.

Access matrix

Administrating the access matrix means defining combinations of user categories such as lead driver or primary controller between which calls are allowed.

In the service, the access matrix is implemented as follows: Role IDs are assigned to combinations of CTs and function codes. Combinations of role IDs of Aparty and B-party within a call are then assigned as being allowed or forbidden.

In the case of call forwarding, i.e. if a call is redirected from a B-party to a C-party, the combination of the role IDs of B-party and C-party determine whether the call is allowed or forbidden.

Enhanced access matrix

If a combination in the access matrix has the value 4, a enhanced access matrix check has to be performed to determine whether a call is allowed. Administrating the enhanced access matrix means defining combinations of enhaced Role IDs between which calls are allowed.

Rules

For the enhanced access matrix, the following rules apply:

  • The format of the MSISDNs must be <iec><cc><ndc><sn> if the MsisdnWithIec flag is set to true. Otherwise, the format has to be <cc><ndc><sn>.
  • If the A-party has registered to an FN, his FN is searched in the enhanced access matrix.
  • A call is allowed if no enhanced role is found for A-party or B-party.
  • If the B-Party is of CT 8 and if the CdPa is not contained in the Number field of the Enhanced Role Assignment data set, the call is not allowed.
  • The Functions Flag and the Cabradio Flag are only checked if the MSISDN of the CdPa is of CT 8.
  • For FNs of CT 1-7, the Functions Flag is always set to true.
  • For FNs beginning with 9, the Functions Flag is always set to false.
