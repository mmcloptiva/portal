Functional Description

Introduction

This section provides a functional description of the deregistration mechanisms via USSD.

Deregistration

The following table provides an overview of how USSD deregistration is handled:

Table 1. USSD deregistration
Stage Description
1 A user sends a USSD string containing the deregistration information via the HLR to .
2 The service searches the Function Register data set for the FN to be deregistered.
3 In the case of normal deregistration the service compares the MSISDN in Function Register to the CgPa, and continues if both numbers match. In the case of forced deregistration, the service compares the MSISDN in the deregistration string with the MSISDN in the Function Register and continues if they match,
4 The service checks if FN in question can be deregistered (Deregistration Allowed flag in Call Type data set).
5 The MSISDN (Msisdn) and the expiration date (Expiration Date) are deleted from the Function Register data set.
6 If enhanced automatic conference is used, the group deregistration procedure is started (see Enhanced Automatic Conference).
7 In the case of forced deregistration, a USSD string is sent (containing a string specified in Ussd Additional Info Field) to the deregisterd user. If an error occurs when the message is sent, the sending of the message is repeated Max Repetition USSD Notification times. The time interval after which a newUSSD notification attempt is started if the previous attempt was unsuccessful, is defined in the Ussd Repetition Interval field.

Bulk deregistration

When the system detects an incoming deregistration request, it checks whether the request contains a BULK_TAG. If yes, it checks if bulk operations are allowed, if the request is correctly formatted and if it was sent by a lead driver or a cab radio. Only if all checks are successful, the deregistration is performed (separately for each FC as described for normal deregistration).

Note: If other features interworking with an external system are implemented as well (e.g., external databases or a standby system in the case of geographical redundancy), one bulk operation will cause up to 10 single messages to each of the external systems.

Delayed deregistration at border crossing

The network operator can enable/disable this feature generally (Is Delayed Deregistration Allowed).

When the train crosses a border, the system receives an automatic deregistration message. The system sends a confirmation message to the cab radio or the mobile phone right away. If the feature is enabled and the deregistration message was sent by a cab radio, the system removes the old FN-MSISDN assignment from the system data base only later. This delay period can be administrated (DDRTimeOffset). Thus, anyone dialling the old FN is still able to contact the “old” MSISDN. If during the delay period, another cab radio registers to the FN, accepts and confirms the registration, and, thenceforward, uses the new FN-MSISDN mapping.

The system automatically calculates an expiration date for the delayed deregistration (Expiration Date For Delayed Deregistration) by adding the delay period (Delayed Deregistration Time Offset) to the deregistration time. The Expiration Date For Delayed Deregistration parameter should not be modified.
